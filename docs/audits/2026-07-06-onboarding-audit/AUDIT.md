# TokenTrail 普通用户接入体验审查

日期：2026-07-06

## 审查范围

本次只审查一个核心任务：一个第一次接触 TokenTrail 的普通用户，能否顺利安装、识别自己的 AI 工具、完成数据接入，并确认接入成功。

审查证据：

1. `01-dashboard.png`：首次打开 Dashboard。
2. `02-quick-start.png`：接入指南的“快速开始”。
3. `03-openclaw.png`：OpenClaw 接入说明。
4. `04-codex.png`：Codex 接入说明。

同时核对了 README、接入文档、同步逻辑和接入指南实现。

## 总体结论

TokenTrail 已经是一个对开发者有价值、接入后也比较好用的本地用量面板，但目前还不是普通用户可以轻松上手的产品。

- 接入后的日常使用：8/10
- 熟悉终端的开发者首次接入：5/10
- 普通用户首次接入：3/10

最大的阻力不是 Dashboard，而是每个数据源都要求用户理解不同的采集机制。当前产品把这种底层差异直接暴露给用户了。

## 分步健康度

### Step 1：打开 Dashboard

健康度：良好

优点：

- 系统状态、来源健康、最近同步和记录数都可见，接入后容易判断工具是否在工作。
- “接入指南”入口位于顶部，能够被发现。
- 本地存储和备份状态能建立信任。

问题：

- 页面默认展示已有数据，但新用户没有数据时缺少明确的首要行动，例如“检测我的工具”或“开始接入”。
- 来源健康更像运维状态，没有告诉用户“哪些工具已检测、哪些还没接入、下一步该做什么”。

### Step 2：打开快速开始

健康度：勉强可用

优点：

- 有分步说明和复制按钮。
- 将启动、CLI、Skill、同步集中到一个位置，比只放 README 更好。

问题：

- 用户已经打开 Dashboard，却又看到“在项目目录启动服务”，上下文不一致。
- 第一屏要求理解项目目录、CLI、Skill、命令行和同步，认知负担过高。
- “安装 Skill”不是采集用量的必要条件，但在主流程里容易被误解为必须步骤。
- 没有自动检测、执行按钮、进度状态或成功确认；用户复制命令后必须自己判断是否做对。

### Step 3：接入 OpenClaw / Hermes

健康度：不适合普通用户

优点：

- 明确说明必须使用真实 `response.usage`，数据原则是正确的。
- JSONL 作为内部统一格式简单、可调试、容易去重。

问题：

- 用户需要修改 OpenClaw/Hermes 的模型调用代码，在响应完成后自行写文件。这已经属于开发工作，不是普通用户配置。
- 页面展示的 `echo` 命令只会写入一条固定示例数据，并不能完成持续接入；用户容易误以为复制执行后就接好了。
- 示例数据会进入真实数据库，可能污染统计。
- 页面没有解释应该到 OpenClaw/Hermes 的哪个配置文件、Hook 或插件位置完成接入。
- 用户看不到接入是否持续生效，只能等待下一次同步后观察记录。

### Step 4：接入 Codex / Claude Code

健康度：基础能力好，表达仍偏技术

优点：

- 本地日志扫描是真正低门槛的方案，不需要修改 Codex 或 Claude Code。
- 不改原始会话文件，隐私和安全边界清楚。

问题：

- “零配置”仍要求用户先确认隐藏目录，再执行同步命令。
- 同一页同时给出三种方式和采集字段，普通用户不知道只需要第一种。
- 应该直接显示“已检测到 Codex，发现 N 个会话”，再提供一个“立即导入”按钮。

## 结构性问题

### 1. 接入模型不统一

当前存在多套心智模型：

- Codex / Claude Code：扫描本地日志。
- TraeWork：扫描历史文件，但接入指南没有 TraeWork 入口。
- OpenClaw / Hermes：要求工具主动写 JSONL。
- 其他工具：HTTP API、CLI、SDK 或代理。

这些实现差异对开发者合理，但不应该成为普通用户必须先理解的产品结构。

### 2. 产品把“接口文档”当成“接入向导”

现在的向导主要提供命令和字段说明。真正的接入向导应该回答：

1. 你电脑上有哪些 AI 工具？
2. 哪些已经检测到？
3. 哪些可以自动接入？
4. 哪些需要一次授权或配置？
5. 是否已经收到第一条真实数据？

### 3. 缺少成功闭环

用户执行命令后，没有明确的“已接入”状态、测试记录、最后心跳或错误恢复建议。复制成功不等于接入成功。

### 4. 安装仍偏源码项目

首次安装需要 Git、Node.js、终端、仓库目录和 npm。普通用户很难独立完成。当前更像开源开发者工具，而不是开箱即用的桌面产品。

## 可访问性风险

以下是从当前实现中可确认或高度可疑的风险，不能据此声称完整 WCAG 合规：

- 接入弹窗缺少明确的 `role="dialog"`、`aria-modal="true"` 和标题关联。
- 弹窗打开后没有看到初始焦点、焦点锁定和关闭后的焦点恢复逻辑。
- 标签页使用普通按钮，但缺少 tablist/tab、`aria-selected` 和键盘方向键语义。
- 标签、命令说明和页脚大量使用 11px 字号，暗色主题下对低视力用户仍偏小。
- 关闭按钮约 32px，低于常见的 44px 触控目标建议。

## 建议优先级

### P0：把接入改成统一的“检测 → 连接 → 验证”

新增一个真正的首次接入向导：

1. 自动检测 Codex、Claude Code、TraeWork、OpenClaw、Hermes。
2. 每个工具显示“已检测 / 未安装 / 待接入 / 已连接 / 有错误”。
3. 自动扫描型工具提供“立即导入”。
4. 可配置型工具提供“一键配置”或生成明确的工具专属配置。
5. 收到第一条真实用量后显示“接入成功”。

### P1：为每个工具封装适配器

对用户只暴露一个入口，例如：

```bash
npx tokentrail connect
```

内部再根据工具选择扫描、Hook、代理或 API。高级用户仍可展开查看 JSONL/API 细节，但不让它们占据主流程。

### P1：停止把示例数据写入当成接入方式

OpenClaw/Hermes 页面中的 `echo` 示例应改成以下二选一：

- 明确标注“仅测试，会写入一条示例记录”；
- 更好的是提供真实适配器安装命令，并在完成后验证最近一条真实调用。

### P2：重做接入页面的信息架构

主流程只保留：

- 自动检测到的工具。
- 推荐方式。
- 一个主按钮。
- 当前状态和错误恢复。

API、CLI、JSONL 字段、去重规则全部放到“高级接入”。

### P2：降低安装门槛

短期可以提供一个安装命令；长期更适合做 macOS 安装包或菜单栏应用。至少不应要求普通用户先克隆仓库再理解 npm 生命周期。

## 推荐的产品定位判断

当前版本适合定位为：

> 面向熟悉终端的 AI 工具重度用户和开发者的本地用量追踪器。

如果目标是普通用户，则下一阶段最值得投入的不是继续增加 Dashboard 模块，而是把“接入不同工具”收敛成一个统一、可验证的安装体验。

## 证据限制

- 本次没有在一台全新、未安装 TokenTrail 的电脑上执行完整安装，因此安装耗时、Node/Git 缺失、权限弹窗和失败恢复仍需单独测试。
- 没有实际修改 OpenClaw/Hermes 的运行配置，因此未验证持续上报链路。
- 可访问性结论基于截图和实现代码，不等同于完整的键盘、读屏器和对比度合规测试。
